1. Project Overview

This project is a dynamic quiz system built with Laravel. Admins can create quizzes, add questions of five different types, and students can take those quizzes and get auto-scored.

What the app can do

  • Quiz CRUD — create, edit, delete quizzes (title + description).
  • Add questions with rich body text, optional image, optional video URL, and a marks value.
  • Five question types: Binary (Yes/No), Single Choice, Multiple Choice, Number Input (with tolerance), Text Input (with match modes).
  • Take a quiz — student sees questions, submits answers, gets an auto-graded result.
  • Review results — per-question marks awarded.

The two big ideas you must explain

Interview Headline #1

Strategy Pattern via a Type Registry. Different question types behave differently. Instead of giant if/else chains scattered across the controller, each type is a class implementing a common QuestionType interface. A registry maps a slug like "binary" to the right handler. Adding a new type = one new class + one line in the service provider.

Interview Headline #2

Single-table polymorphism with a JSON config column. Rather than 5 different question tables, all questions live in one questions table. Type-specific settings (e.g. tolerance for a number, match mode for text) live in a flexible JSON config column. Simple schema, infinite extensibility.

2. The Big Picture: How a Laravel App Works

Before diving into code, let's understand what happens between "user clicks a link" and "user sees a page". Every Laravel app follows the same lifecycle.

The five-step lifecycle

1
Browser sends an HTTP request to your server (e.g. GET /quizzes).
2
Laravel's router looks at the URL + verb and decides which controller method to call.
3
The controller method runs — it talks to the database via the ORM, applies business logic, and prepares data.
4
The controller returns a response — usually by rendering a Blade view (HTML) or redirecting.
5
The browser receives the HTML and renders it.
Mental Model

Think of Laravel as a receptionist. The browser walks in saying "I want page X". The receptionist (router) checks a list, finds the right specialist (controller), the specialist fetches what's needed from the filing cabinet (database), assembles a document (view), and hands it back to the visitor.

The MVC pattern

Laravel is built around MVC — Model, View, Controller:

LetterStands forIn this projectPlain English
MModelapp/Models/Quiz.php, Question.php, etc.The data and rules about it (a Quiz has a title, a Quiz has many Questions).
VViewresources/views/quizzes/*.blade.phpThe HTML the user sees.
CControllerapp/Http/Controllers/QuizController.phpThe traffic cop. Receives the request, asks the model for data, sends data to the view.

3. What is Laravel?

Laravel is a PHP framework. A framework is a collection of pre-written code that solves common problems so you don't have to. Routing, database access, templating, validation, file uploads, sessions, security — all included.

Why Laravel exists

Without a framework, building a web app means you write everything yourself: parsing HTTP requests, escaping HTML, querying the database safely, handling cookies, validating forms. Tedious and error-prone. Laravel gives you tested, secure defaults so you focus on your application's logic.

4. What is an ORM? (Eloquent Explained)

ORM = Object-Relational Mapping. It's a layer that lets you treat database rows as PHP objects, instead of writing raw SQL.

Without an ORM (raw SQL)

$pdo = new PDO("sqlite:database.sqlite");
$stmt = $pdo->prepare("SELECT * FROM quizzes WHERE id = ?");
$stmt->execute([5]);
$row = $stmt->fetch(PDO::FETCH_ASSOC);
echo $row['title'];

With Eloquent (the ORM Laravel uses)

$quiz = Quiz::find(5);
echo $quiz->title;

Same result, much cleaner. Eloquent runs the SQL behind the scenes.

What Eloquent gives you

  • Models — one PHP class per table. Quiz ↔ quizzes table.
  • Relationships — express "a Quiz has many Questions" in code; Eloquent figures out the foreign keys.
  • Query builder — chainable, expressive: Quiz::where('title', 'Math')->latest()->get().
  • Mass assignment protection — only fields in $fillable can be bulk-set.
  • Casts — auto-convert DB values (e.g. JSON column → PHP array; 0/1 → true/false).
  • Eager loading — solve the "N+1 query" problem with one line.

The convention magic

Eloquent uses convention over configuration. By default:

  • Class Quiz → table quizzes (auto-pluralize).
  • Primary key is id.
  • Foreign key for "belongs to Quiz" is quiz_id.
  • Timestamps created_at and updated_at auto-managed.

You can override any of these, but if you follow the convention, you write almost no configuration.

5. The Request Lifecycle in Detail

Let's trace what happens when a user opens http://localhost:8000/quizzes in their browser.

1
Browser fires GET /quizzes. The browser opens a TCP connection to your server.
2
Web server (PHP built-in via php artisan serve) hands the request to public/index.php. This is the only PHP file the web server ever runs — it's the front controller.
3
Laravel boots. The framework loads, reads .env, registers all service providers (including QuestionTypeServiceProvider which sets up our registry).
4
The Router runs middleware and matches the route. Middleware are layers like authentication, CSRF protection, session handling. After they pass, it finds Route::resource('quizzes', QuizController::class) which maps GET /quizzes → QuizController@index.
5
Laravel constructs the controller. Looking at the constructor, if it needs anything (like QuestionTypeRegistry), the service container builds and injects it.
6
index() executes. It runs Quiz::withCount('questions')->latest()->get() which Eloquent translates into SQL and runs against SQLite.
7
The view renders. view('quizzes.index', compact('quizzes')) compiles resources/views/quizzes/index.blade.php with the $quizzes variable.
8
HTML is returned. Laravel sends the rendered HTML back through any "after" middleware, then to the browser.
9
Browser renders. Done.
Likely Question

"What happens between the URL and the page?" — Walk them through these 9 steps. Even high-level acknowledgement of front-controller, service container boot, middleware chain, route match, controller dispatch, view render is impressive.

6. Routing Explained

Routes live in routes/web.php. They tell Laravel: "When the browser asks for THIS URL with THIS method, run THAT controller method."

The actual file

Route::redirect('/', '/quizzes');

Route::resource('quizzes', QuizController::class);

Route::post('quizzes/{quiz}/questions', [QuestionController::class, 'store'])
    ->name('questions.store');
Route::get('questions/{question}/edit', [QuestionController::class, 'edit'])
    ->name('questions.edit');
Route::put('questions/{question}', [QuestionController::class, 'update'])
    ->name('questions.update');
Route::delete('questions/{question}', [QuestionController::class, 'destroy'])
    ->name('questions.destroy');

Route::get('quizzes/{quiz}/take', [AttemptController::class, 'start'])
    ->name('attempts.start');
Route::post('attempts/{attempt}/submit', [AttemptController::class, 'submit'])
    ->name('attempts.submit');
Route::get('attempts/{attempt}', [AttemptController::class, 'show'])
    ->name('attempts.show');

Line by line

Route::redirect('/', '/quizzes');

If anyone visits the root URL /, send them to /quizzes. Saves writing a controller for a one-liner redirect.

Route::resource('quizzes', QuizController::class);

This one line creates 7 routes — the standard RESTful set:

VerbURIMethodNamePurpose
GET/quizzesindexquizzes.indexList all quizzes
GET/quizzes/createcreatequizzes.createShow "new quiz" form
POST/quizzesstorequizzes.storeSave the new quiz
GET/quizzes/{quiz}showquizzes.showShow one quiz
GET/quizzes/{quiz}/editeditquizzes.editShow edit form
PUT/PATCH/quizzes/{quiz}updatequizzes.updateSave edits
DELETE/quizzes/{quiz}destroyquizzes.destroyDelete the quiz

QuizController::class

This is not creating an instance. ::class is a PHP language construct that returns the class name as a string: "App\Http\Controllers\QuizController". We use it instead of writing the string ourselves because the IDE can autocomplete and refactor it safely.

Named routes — ->name('questions.store')

A label for the route. Anywhere in the app you can write route('questions.store', $quiz) instead of hardcoding the URL. If the URL ever changes, every reference still works.

Route parameters — {quiz}, {question}, {attempt}

Anything in curly braces is a placeholder. Laravel passes the value to the controller. With route-model binding (see below), Laravel goes a step further: if the controller method declares Quiz $quiz, Laravel looks up the model by ID and passes the actual Quiz object — or returns 404 if not found.

Route-Model Binding (must know!)

When a controller method signature says public function show(Quiz $quiz), Laravel sees the URL /quizzes/5, runs Quiz::findOrFail(5), and hands the resulting model object to your method. You never write the lookup yourself. If 5 doesn't exist, the user gets a 404 automatically.

Why some routes are written manually instead of Route::resource

The "questions" routes are nested under quizzes (/quizzes/{quiz}/questions) and only some of the 7 RESTful actions are needed. Easier to write 4 lines than configure a partial nested resource.

7. Controllers in Detail

A controller is a class with public methods. Each method handles one type of HTTP request. The method's job is: read the request, do work (usually DB), return a response.

7.1 QuizController

index() — list quizzes

public function index()
{
    $quizzes = Quiz::withCount('questions')->latest()->get();
    return view('quizzes.index', compact('quizzes'));
}
  • Quiz::withCount('questions') — fetch all quizzes AND for each one, count its questions (in a single SQL query, not N+1). Each quiz then has a questions_count integer property.
  • ->latest() — order by created_at DESC, newest first.
  • ->get() — execute the query, return a Collection of Quiz objects.
  • view('quizzes.index', compact('quizzes')) — render resources/views/quizzes/index.blade.php and pass $quizzes into it. compact('quizzes') is shorthand for ['quizzes' => $quizzes].

create() — show "new quiz" form

public function create()
{
    return view('quizzes.create');
}

Renders the empty form. No DB work needed.

store() — save a new quiz

public function store(Request $request)
{
    $data = $request->validate([
        'title' => ['required', 'string', 'max:255'],
        'description' => ['nullable', 'string'],
    ]);

    $quiz = Quiz::create($data);

    return redirect()->route('quizzes.edit', $quiz)
        ->with('status', 'Quiz created. Add your questions below.');
}
  • Validate first. If validation fails, Laravel auto-redirects back with errors. If it passes, $data contains only the validated fields.
  • Quiz::create($data) — create a new row from the array (only fields in $fillable on the model are accepted, for security).
  • Redirect to the edit page, not back to a "show" page. Why? Because creating a quiz is just step 1 — you still need to add questions. Flash a one-time success message via ->with('status', ...).

edit() — show the editor

public function edit(Quiz $quiz, QuestionTypeRegistry $registry)
{
    $quiz->load('questions.options');

    return view('quizzes.edit', [
        'quiz' => $quiz,
        'types' => $registry->options(),
        'registry' => $registry,
    ]);
}
  • Two things injected: Quiz $quiz (route-model binding from the URL) and QuestionTypeRegistry $registry (from the service container).
  • $quiz->load('questions.options') — eager-load questions AND each question's options. Without this, the view would trigger one extra query per question (the N+1 problem).
  • Three variables to the view: the quiz itself, a slug→label map for the question-type dropdown, and the full registry object so the view can ask for type-specific partials.

update() — save quiz edits

public function update(Request $request, Quiz $quiz)
{
    $data = $request->validate([
        'title' => ['required', 'string', 'max:255'],
        'description' => ['nullable', 'string'],
    ]);

    $quiz->update($data);

    return redirect()->route('quizzes.edit', $quiz)->with('status', 'Quiz updated.');
}

Same pattern as store, but updating an existing model. $quiz->update($data) sets the new values and saves.

destroy() — delete a quiz (and clean up files)

public function destroy(Quiz $quiz)
{
    foreach ($quiz->questions as $question) {
        if ($question->image_path) {
            Storage::disk('public')->delete($question->image_path);
        }
        foreach ($question->options as $option) {
            if ($option->image_path) {
                Storage::disk('public')->delete($option->image_path);
            }
        }
    }

    $quiz->delete();

    return redirect()->route('quizzes.index')->with('status', 'Quiz deleted.');
}
Why the manual file cleanup?

The DB rows cascade-delete via foreign key constraints (cascadeOnDelete in the migration). But uploaded files on disk are not tracked by Eloquent — they would be orphaned forever. So we delete them manually before the row goes.

7.2 QuestionController

This controller manages individual questions. Critically, it uses the QuestionTypeRegistry to delegate type-specific work — it never has an if/else over question types.

Constructor

public function __construct(private QuestionTypeRegistry $registry) {}

Constructor property promotion (PHP 8). Laravel injects the registry on every request. The controller never knows about specific type classes.

store() — create a question

public function store(Request $request, Quiz $quiz)
{
    $base = $this->validateBase($request);
    $type = $this->registry->for($base['type']);

    $imagePath = $request->hasFile('image')
        ? $request->file('image')->store('questions', 'public')
        : null;

    $question = $quiz->questions()->create([
        'type' => $base['type'],
        'body_html' => $base['body_html'],
        'image_path' => $imagePath,
        'video_url' => $base['video_url'],
        'marks' => $base['marks'],
        'position' => $quiz->questions()->max('position') + 1,
        'config' => $type->buildConfig($request->all()),
    ]);

    $type->persistOptions($question, $request->all());
    $this->persistOptionImages($question, $request);

    return redirect()->route('quizzes.edit', $quiz)->with('status', 'Question added.');
}

Step by step:

  1. Validate common fields via validateBase() (body, marks, video, image, type).
  2. Look up the type handler from the registry — $type is now a BinaryType, MultipleChoiceType, etc.
  3. If an image was uploaded, store it on the public disk and capture the relative path.
  4. Create the question via the relationship — $quiz->questions()->create([...]) auto-fills quiz_id.
  5. Position = current max + 1 → new questions go at the end.
  6. config = whatever the type handler returns from buildConfig(). Stored as JSON in the DB.
  7. Delegate option saving — the type knows how (multiple choice creates 4 rows; binary creates 0).
  8. Handle option images separately.
  9. POST/Redirect/GET — redirect to the editor with a flash message.

edit() — show edit form

public function edit(Question $question)
{
    $question->load('options');
    return view('questions.edit', [
        'question' => $question,
        'type' => $this->registry->for($question->type),
    ]);
}

Pass the question and its type handler to the view. The view uses the handler's editorView() method to include the right Blade partial.

update()

Same pattern as store but with extra image logic for three cases:

  • New image uploaded → delete old, store new.
  • "Remove image" checkbox checked → delete file, null the column.
  • Neither → leave the existing image alone.

destroy()

Captures $quizId first (we need it for the redirect after deletion), deletes all uploaded files, then deletes the question. Option rows cascade via FK constraint.

validateBase() — DRY validation helper

private function validateBase(Request $request, bool $isUpdate = false): array
{
    $rules = [
        'body_html' => ['required', 'string'],
        'video_url' => ['nullable', 'url'],
        'marks' => ['required', 'integer', 'min:1'],
        'image' => ['nullable', 'image', 'max:4096'],
    ];

    if (! $isUpdate) {
        $rules['type'] = ['required', 'string', 'in:'.implode(',', array_keys($this->registry->all()))];
    }

    return $request->validate($rules);
}
Clever bit

The type validation rule is generated from the registry: in:binary,single_choice,multiple_choice,number,text. Add a new type → validation auto-extends. Open/Closed Principle in action.

persistOptionImages()

Form sends option images as an array (option_images[0], option_images[1], ...). For each option, check if there's a file at the same index, replace if so.

7.3 AttemptController

Handles a student taking a quiz. Three actions: start, submit, show.

start() — begin an attempt

public function start(Quiz $quiz)
{
    abort_if($quiz->questions()->count() === 0, 404, 'This quiz has no questions yet.');
    $quiz->load('questions.options');

    $attempt = $quiz->attempts()->create([
        'started_at' => now(),
        'max_score' => $quiz->totalMarks(),
    ]);

    return view('attempts.take', [
        'quiz' => $quiz,
        'attempt' => $attempt,
        'registry' => $this->registry,
    ]);
}
  • Guard: 404 if the quiz has no questions.
  • Eager-load questions and options.
  • Create an Attempt record — captures start time and max possible score.
  • Render the "take quiz" page.

submit() — score the answers

public function submit(Request $request, Attempt $attempt)
{
    abort_if($attempt->isSubmitted(), 403, 'This attempt has already been submitted.');

    $attempt->load('quiz.questions.options');
    $raw = (array) $request->input('answers', []);

    $total = 0.0;

    foreach ($attempt->quiz->questions as $question) {
        $type = $this->registry->for($question->type);
        $payload = $type->normalizePayload($raw[$question->id] ?? null);
        $marks = $type->evaluate($question, $payload);

        $attempt->answers()->create([
            'question_id' => $question->id,
            'payload' => $payload,
            'marks_awarded' => $marks,
        ]);

        $total += $marks;
    }

    $attempt->update([
        'submitted_at' => now(),
        'score' => $total,
        'max_score' => $attempt->quiz->totalMarks(),
    ]);

    return redirect()->route('attempts.show', $attempt);
}

The scoring loop is the heart of the app. For every question:

  1. Get the right type handler.
  2. Normalize the raw form input into a clean array.
  3. Evaluate — the type returns marks awarded (0 to $question->marks).
  4. Save an Answer row with the payload and marks.
  5. Accumulate the total.

After the loop, update the Attempt with the final score and submission time.

Notice: zero if/else over question types

The scoring loop is type-agnostic. It just calls methods on the type handler. This is the strategy pattern paying off — the controller would not change if you added 50 more question types tomorrow.

show() — display the result

public function show(Attempt $attempt)
{
    abort_unless($attempt->isSubmitted(), 404);
    $attempt->load('quiz', 'answers.question.options');

    return view('attempts.result', [
        'attempt' => $attempt,
        'registry' => $this->registry,
    ]);
}

Eager-load everything needed for the review page; render the result view.

The Quiz model

class Quiz extends Model
{
    use HasFactory;

    protected $fillable = ['title', 'description'];

    public function questions(): HasMany
    {
        return $this->hasMany(Question::class)->orderBy('position');
    }

    public function attempts(): HasMany
    {
        return $this->hasMany(Attempt::class);
    }

    public function totalMarks(): int
    {
        return (int) $this->questions()->sum('marks');
    }
}
  • $fillable — only these fields can be mass-assigned via create() or update(). Security against attackers POSTing extra fields.
  • questions() — defines the has-many relationship. The orderBy('position') means whenever you load questions, they come out in order.
  • totalMarks() — a helper to sum marks across all questions. Used when starting/submitting an attempt.

The Question model

class Question extends Model
{
    protected $fillable = ['quiz_id', 'type', 'body_html', 'image_path',
                           'video_url', 'marks', 'config', 'position'];

    protected $casts = [
        'config' => 'array',
        'marks' => 'integer',
        'position' => 'integer',
    ];

    public function quiz(): BelongsTo { return $this->belongsTo(Quiz::class); }
    public function options(): HasMany { return $this->hasMany(Option::class)->orderBy('position'); }
}
The casts array — important!

The config column stores JSON in the DB. With 'config' => 'array' in $casts, Eloquent automatically decodes it into a PHP array when you read, and encodes it back when you save. You never call json_encode or json_decode yourself.

The migration that creates questions

Schema::create('questions', function (Blueprint $table) {
    $table->id();
    $table->foreignId('quiz_id')->constrained()->cascadeOnDelete();
    $table->string('type');
    $table->longText('body_html');
    $table->string('image_path')->nullable();
    $table->string('video_url')->nullable();
    $table->unsignedInteger('marks')->default(1);
    $table->json('config')->nullable();
    $table->unsignedInteger('position')->default(0);
    $table->timestamps();
    $table->index(['quiz_id', 'position']);
});
  • foreignId('quiz_id')->constrained()->cascadeOnDelete() — creates a foreign key to quizzes.id. If the parent quiz is deleted, all its questions are auto-deleted by the database.
  • json('config') — native JSON column type.
  • index(['quiz_id', 'position']) — composite index because we frequently query questions for a quiz, ordered by position. Speeds up that query massively.

What is a migration?

A migration is a versioned, code-based way to evolve the database schema. Instead of running raw CREATE TABLE SQL by hand, you write a PHP file and run php artisan migrate. Every team member, every server, gets the same schema. You can also migrate:rollback to undo.

9. Eloquent in Practice

The query patterns used in this app

Quiz::withCount('questions')->latest()->get()

Fetches all quizzes, attaches a questions_count integer to each (in one efficient subquery), orders by created_at desc.

$quiz->load('questions.options')

Eager-load: fetch all questions for this quiz, AND all options for each question. Three queries total instead of N+1. The dot notation means "nested relationship".

$quiz->questions()->create([...])

Create a new Question that belongs to this quiz. Eloquent auto-sets quiz_id = $quiz->id. You don't pass it.

$quiz->questions()->max('position') + 1

Run a SQL MAX(position) on this quiz's questions. Used to set the position of the next question to "last + 1".

$question->options()->delete()

Delete every option belonging to this question — used when re-saving options on update.

$question->options->firstWhere('id', $pickedId)

This is on the collection, not the query. Once options are loaded into memory, find the one with that id without hitting the DB again.

The N+1 problem (very common interview question)

N+1 Defined

You fetch N parents in 1 query. Then you loop and access a relationship on each — triggering N extra queries. Total: N+1 queries. For 100 quizzes with questions, that's 101 SQL hits per page load.

Fix: Eager-load with with() or load(). Or use withCount() when you only need the count.

load vs loadMissing vs fresh

MethodBehavior
$model->load('rel')Always loads the relationship (re-runs the query)
$model->loadMissing('rel')Loads only if not already loaded
$model->fresh('rel')Returns a NEW model freshly fetched from DB with that relationship

10. The Question Type System (Strategy + Registry)

This is the most important architectural piece. Be ready to walk through it.

The problem it solves

Without this pattern, every method that touches questions would have a giant if/else over the type:

// THE MESS THIS CODE AVOIDS
if ($question->type === 'binary') { /* score binary */ }
elseif ($question->type === 'multiple_choice') { /* score MC */ }
elseif ($question->type === 'number') { /* score number */ }
// ... in store(), update(), submit(), every view ...

Adding a new type would mean editing 6+ files. Forgetting one breaks something silently.

The solution

Each type is a class implementing a QuestionType interface. A registry holds them. The controller looks up by slug and delegates.

The interface (the contract)

interface QuestionType
{
    public function slug(): string;
    public function label(): string;
    public function buildConfig(array $input): array;
    public function persistOptions(Question $q, array $input): void;
    public function normalizePayload(mixed $raw): array;
    public function evaluate(Question $q, array $payload): float;
    public function editorView(): string;
    public function inputView(): string;
    public function reviewView(): string;
}
MethodPlain English
slug()"What's my unique short name?"
label()"What's my human-readable name (for UI)?"
buildConfig($input)"What type-specific settings should we save in the JSON config?"
persistOptions($q, $input)"Save my answer options however I see fit."
normalizePayload($raw)"Clean up the raw user answer into a structured array."

The registry

class QuestionTypeRegistry
{
    private array $types = [];

    public function register(QuestionType $type): self
    {
        $this->types[$type->slug()] = $type;
        return $this;
    }

    public function for(string $slug): QuestionType
    {
        if (! isset($this->types[$slug])) {
            throw new RuntimeException("Unknown question type: {$slug}");
        }
        return $this->types[$slug];
    }

    public function all(): array { return $this->types; }

    public function options(): array
    {
        $out = [];
        foreach ($this->types as $slug => $type) {
            $out[$slug] = $type->label();
        }
        return $out;
    }
}

Tiny class. Four methods:

  • register — add a type. Returns $this for chaining.
  • for($slug) — look up; throw if missing (fail fast).
  • all() — return the whole map (used to build the validation rule dynamically).
  • options() — return slug→label for dropdowns.

How it gets populated — the service provider

class QuestionTypeServiceProvider extends ServiceProvider
{
    public function register(): void
    {
        $this->app->singleton(QuestionTypeRegistry::class, function () {
            return (new QuestionTypeRegistry)
                ->register(new BinaryType)
                ->register(new SingleChoiceType)
                ->register(new MultipleChoiceType)
                ->register(new NumberType)
                ->register(new TextType);
        });
    }
}
What this means

"Hey Laravel, when anyone asks for a QuestionTypeRegistry, run this closure once, build a registry pre-populated with all 5 types, and cache it for the rest of the request." That's what singleton does.

11. The Service Container & Dependency Injection

What is the service container?

Laravel's service container is a giant PHP factory. You can ask it: "Give me an instance of class X." It will:

  1. Look at the class's constructor.
  2. For each parameter, recursively ask the container for that type.
  3. Build the whole tree of dependencies and return the assembled object.

Why this matters here

Look at the controller:

public function __construct(private QuestionTypeRegistry $registry) {}

You never write new QuestionController(new QuestionTypeRegistry()). Laravel does it for you, every request. Decoupled and testable.

What is dependency injection?

Instead of a class building its own dependencies ($this->db = new Database()), they're passed in. The class doesn't care where they came from. In tests, you can pass mocks instead of real things.

Singleton vs bind

MethodBehavior
$this->app->singleton(X, fn)Run fn once per request, cache result. Same instance every time.
$this->app->bind(X, fn)Run fn every time X is requested. Fresh instance each time.

The registry is a singleton — no need to rebuild it five times in one request.

12. File Uploads & Storage

The upload flow

$imagePath = $request->hasFile('image')
    ? $request->file('image')->store('questions', 'public')
    : null;
  • $request->hasFile('image') — was a file uploaded under the image input name?
  • $request->file('image') — get the UploadedFile object.
  • ->store('questions', 'public') — save it under the questions/ folder on the public disk. Returns the relative path with a random filename.

What is "the public disk"?

Laravel's filesystem is configured with named "disks" (in config/filesystems.php). The public disk maps to storage/app/public, which is symlinked to public/storage via php artisan storage:link. This makes uploaded files accessible at URLs like /storage/questions/abc123.jpg.

Why store the path, not the file?

Files belong on disk. The DB just stores a string path so we know where to find them. Tiny rows, fast queries.

Cleanup

When updating or deleting, the controller manually removes old files via Storage::disk('public')->delete($path). The DB doesn't track files, so we have to.

15. The Complete Request Walkthrough

Let's trace one full user journey: a student submits a quiz attempt.

Setup

Quiz #5 has 3 questions: a binary, a multiple choice, and a number. Student starts the attempt, answers them, hits Submit.

Step 1: Browser sends the request

POST /attempts/12/submit
Content-Type: application/x-www-form-urlencoded
Body: answers[1]=yes&answers[2][]=4&answers[2][]=5&answers[3]=42

Step 2: Laravel boots and routes

Router finds: Route::post('attempts/{attempt}/submit', [AttemptController::class, 'submit']).

Step 3: Controller is built

Container sees AttemptController needs a QuestionTypeRegistry. It gets the cached singleton (already populated with 5 types).

Step 4: Route-model binding

Laravel sees {attempt} = "12" and the method signature is Attempt $attempt. It runs Attempt::findOrFail(12) and passes the model.

Step 5: Guard against double-submit

abort_if($attempt->isSubmitted(), 403, ...) — if already submitted, return 403.

Step 6: Eager-load the data we need

$attempt->load('quiz.questions.options') — three nested levels in one go.

Step 7: Loop and score

For each question:

  • Question 1 (binary): $type = registry->for('binary'). Normalize: ['value' => 'yes']. Evaluate against config['correct']. Save Answer with marks.
  • Question 2 (MC): Normalize: ['option_ids' => [4, 5]]. Evaluate (strict or partial). Save.
  • Question 3 (number): Normalize: ['value' => 42.0]. Evaluate against expected ± tolerance. Save.

Step 8: Update the attempt with totals

$attempt->update([
    'submitted_at' => now(),
    'score' => $total,
    'max_score' => $attempt->quiz->totalMarks(),
]);

Step 9: Redirect

return redirect()->route('attempts.show', $attempt); — sends a 302 redirect to /attempts/12.

Step 10: Browser follows the redirect

New GET /attempts/12 request → show() method → renders the result page with each question, the student's answer, and marks awarded. Done.

17. Interview Questions with Answers

Project & Architecture

Walk me through this project.
It's a dynamic quiz app in Laravel. Admins create quizzes and add questions of 5 different types — binary, single choice, multiple choice, number, text. Students take quizzes and get auto-graded. The interesting design is the strategy pattern via a type registry: each question type is its own class implementing a common interface; the controller never has an if/else over types. All questions live in one table with a JSON config column for type-specific settings.
Why didn't you use polymorphic models or one table per question type?
Polymorphic models work but add complexity (morphTo, morphMany, two columns per relation). Separate tables per type would mean joining 5 tables and complicate the editor and scoring. Single table with a JSON config column is the simplest design that supports unbounded types. The trade-off is you can't easily query into JSON in vanilla SQL, but for read-with-the-row settings, it's fine.
How would you add a new question type tomorrow?
Three steps: (1) Create a class implementing QuestionType with the 9 interface methods. (2) Add one line in QuestionTypeServiceProvider to register it. (3) Create the 3 Blade partials for editor / input / review. No controller changes. Validation auto-extends because the rule is built from the registry.

Laravel Basics

What happens between the URL and the page?
Browser sends HTTP request → web server passes to public/index.php → Laravel boots, registers service providers → middleware runs → router matches the route → controller is built (DI) → controller method runs (talks to DB, validates) → returns a view or redirect → response goes back through middleware → browser renders.
What is MVC?
Model-View-Controller. Model = data + business rules (Eloquent models). View = presentation (Blade templates). Controller = traffic cop that takes requests, asks the model, sends data to the view. Separation of concerns.
What does Route::resource do?
It registers 7 RESTful routes for a resource: index, create, store, show, edit, update, destroy. Saves writing them by hand and enforces convention.
What is route-model binding?
When a URL has a parameter like {quiz} and the controller method declares Quiz $quiz, Laravel automatically runs Quiz::findOrFail(...) with the URL value and passes the model. If not found, it 404s. You don't write the lookup yourself.
What does QuestionController::class mean?
It's a PHP language construct that returns the fully-qualified class name as a string. Used instead of typing the string manually so the IDE can autocomplete and refactor it. It's NOT creating an instance.
What does ->name('something') do on a route?
Gives the route a label so you can reference it via route('something', $params) elsewhere instead of hardcoding the URL. If the URL ever changes, every reference still works.

Eloquent & ORM

What is an ORM and why use one?
Object-Relational Mapping. It lets you treat DB rows as PHP objects instead of writing raw SQL. Cleaner code, less repetition, automatic SQL generation, type safety, easy relationships, protection against SQL injection by default.
What's the N+1 problem and how do you solve it?
You fetch N parent rows in 1 query, then loop and access a relationship on each — triggering N more queries. For 100 quizzes, that's 101 SQL hits. Solve with eager loading: Quiz::with('questions')->get() uses just 2 queries total.
What does withCount do?
Adds a count of related rows as a new property without loading them. Quiz::withCount('questions') attaches questions_count via a single SQL subquery. Efficient.
What is $fillable?
An allow-list of fields that can be mass-assigned via create() or update(). Without it, an attacker could POST extra fields like is_admin=1 and Eloquent would happily save them. Mass assignment protection.
What are casts on a model?
Auto-convert DB values to PHP types. 'config' => 'array' means the JSON column is automatically decoded to a PHP array on read and encoded on write. 'is_correct' => 'boolean' turns the DB's 0/1 into true/false.
What's the difference between hasMany and belongsTo?
hasMany is the parent side (Quiz has many Questions). belongsTo is the child side (Question belongs to Quiz). The foreign key (quiz_id) lives on the child table.
What does $quiz->questions()->create([...]) do that Question::create([...]) wouldn't?
It auto-fills quiz_id from $quiz. The relationship knows the foreign key, so you don't have to set it manually.
When would you use load vs loadMissing vs fresh?
load always queries (use after creation). loadMissing only queries if not already loaded (avoids redundancy). fresh returns a new instance freshly fetched from DB (use after data may have changed under you).

The Type System

Why a registry instead of a switch statement?
A switch couples the controller to every concrete type. Adding a type means editing the switch in every method that uses it. With a registry, the controller depends only on the abstraction. Adding a type is a one-class, one-line change. Open/Closed Principle.
What design pattern is this?
It's the Strategy Pattern — interchangeable algorithms behind a common interface. The registry is a Service Locator for those strategies. The composition is a textbook way to eliminate type-based conditionals.
Where does the registry get populated?
In QuestionTypeServiceProvider::register(). It calls $this->app->singleton to bind the registry as a singleton, with a closure that creates the registry and chains register() calls for all 5 types. Runs once during boot.
Why is the registry a singleton?
No reason to rebuild it multiple times in one request — the type instances are stateless. A singleton means one instance, shared by everyone who asks. Saves work and memory.

Service Container & DI

What is the service container?
Laravel's IoC (inversion of control) container. It builds objects and injects dependencies. You ask it for a class; it inspects the constructor, resolves each parameter recursively, and hands you a fully built instance.
What is dependency injection?
Passing dependencies in via constructor (or method) instead of building them inside the class. Decouples the class from concrete implementations and makes testing easy — pass mocks in tests.
How is the registry injected into the controller?
The constructor declares private QuestionTypeRegistry $registry. When Laravel needs to call the controller, the container builds it, sees the parameter type, looks up the singleton binding, and passes the registry instance.

HTTP & Forms

What is the POST/Redirect/GET pattern?
After a form POST that mutates data, return a redirect to a GET URL instead of rendering directly. This way, refreshing the resulting page doesn't re-submit the form. Standard web hygiene.
What does ->with('status', 'message') do?
Flashes data to the session for the next request only. The view on the redirected-to page can read session('status') to show a one-time success message.
What is CSRF and how does Laravel handle it?
Cross-Site Request Forgery. An attacker tricks a logged-in user into submitting a request. Laravel adds a hidden token to every form (via @csrf) and verifies it on POST/PUT/DELETE via middleware. No matching token → 419 error.
How does file upload work in Laravel?
The form needs enctype="multipart/form-data". The file arrives as an UploadedFile. $request->file('image')->store('folder', 'public') saves it to the public disk and returns the relative path, which you store in the DB.

Validation

How does $request->validate(...) work?
If validation fails, Laravel auto-redirects back with errors and old input flashed to session. If it passes, you get an array of only the validated fields. Anything else is dropped.
Why generate the type validation rule from the registry?
So adding a new type doesn't require updating the validation rule. The rule becomes in:binary,single_choice,multiple_choice,number,text dynamically. One source of truth.

Database

What is a migration?
A versioned PHP file that defines a schema change. php artisan migrate applies them in order. Lets every developer and server have an identical schema, with rollback support.
What does foreignId('quiz_id')->constrained()->cascadeOnDelete() do?
Creates an unsigned bigint quiz_id, adds a foreign key constraint pointing to quizzes.id, and tells the DB to auto-delete this row when the referenced quiz is deleted.
Why a composite index on (quiz_id, position)?
The most common query is "questions for this quiz, ordered by position". A composite index makes both the WHERE and ORDER BY use the index, giving O(log n) lookup instead of full scan + sort.
Why store config as JSON instead of separate columns?
Different question types need different settings. A binary needs {correct}; a number needs {expected, tolerance}; a text needs {expected, match}. Adding columns for every possible setting bloats the schema. JSON gives schema flexibility per type while keeping one table.

Scoring & Logic

How does multiple-choice scoring work?
Two modes via config['scoring']: strict means the picked set must equal the correct set exactly — full marks or zero. partial awards (right − wrong) / total of the marks, floored at 0.
How does number scoring work?
abs(given − expected) ≤ tolerance → full marks; else zero. The tolerance is per-question, set by the admin.
What match modes does text support?
Four: exact (case-sensitive equality), ci (case-insensitive), contains (substring, case-insensitive), regex (regex match). Configurable per question.

Improvements & Trade-offs

What would you improve if you had more time?
(1) Wrap store and submit in DB transactions so partial failures roll back. (2) Add authorization (Policies) — currently anyone with a URL can edit. (3) Add per-type form-request validation classes instead of doing common validation in the controller. (4) Add tests for each type's evaluate method. (5) Add a QuestionType abstract base class to share common boilerplate.
What if scoring becomes more complex (e.g. partial number credit, fuzzy text)?
The architecture handles it: extend the type's evaluate method, add fields to buildConfig, update the editor partial. Nothing else changes.
What if a question type needs custom validation rules?
Add a validationRules(): array method to the QuestionType interface. Each type returns its own rules. The controller merges them with the base rules. Single Responsibility maintained.
